⚙️ Pipeline Architecture & Trade-offs

How the system works and design decisions made

🔧 Pipeline Design

  • Python-based ETL — single-file pipeline (run_pipeline.py) orchestrating ingestion, normalisation, classification, linking, and insight generation
  • 5 ingestion mediums: Excel (openpyxl), CSV (pandas), RSS (feedparser), Web scraping (BeautifulSoup), REST API (requests)
  • Unified JSON schema — every record normalised to: id, source, source_type, category, title, summary, content, entities, metadata, sectors, timestamp, tags, cross_links

🧠 NLP Strategy

  • No LLM / no brute-force AI — keyword-based classification with curated dictionaries for sectors, property types, and locations
  • Regex-based entity extraction — financial values (£/€/$), organisations (capitalisation heuristics), locations (EU city dictionary + CSV enrichment)
  • Cross-source linking — records linked by shared entities (location + org + sector overlap scoring)

⚖️ Trade-offs

  • Fallback data — when external sources (RSS, scraping) fail, representative fallback data ensures the pipeline always produces a complete dataset
  • Shallow NLP vs deep ML — keyword matching trades recall for simplicity and speed; a production system would use NER models
  • Static dashboard — pure HTML/CSS/JS with JSON data; no backend server needed for deployment, enabling instant Vercel/Netlify hosting
